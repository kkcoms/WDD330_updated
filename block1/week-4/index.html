Chapter 8: Forms

Notes

You don't have to only process form information on the backend. It's becoming more common to process information in a form on the front end with JavaScript.

Events

  • The focus event occurs when an element is focused on.

  • The blur event occurs when the user moves the focus away from the form element.

  • The change event occurs when the user moves the focus away from the form elementafter changing it

  • The submit event occurs when the form is submitted.

    "Usually, this will send the content of the form to the server to be processed, but we can use JavaScript to intercept the form before it's sent by adding a submit event listener."

    elements for longer text entry

Chapter 12: OOP

Notes

I can create new objects using the object literal notation or by using a constructor function. A constructor function defines the properties and methods of an object. Alternatively, "ES6 introduced the new class declaration syntax that does exactly the same thing as a constructor function, but looks much similar to writing a class in a class-based programming language."

If I don't know the name of a constructor function or class, I can still make an instance of its object by using the constructor property like this: const greenDice = new redDice.constructor(10); greenDice instanceOf Dice true

"The prototype can be used to add any new properties and methods after the class has been declared. It should be used to define any properties that will remain the same for every instance of the class."

Using mixins in JavaScript is quite different than how I have used them before. The name is a good description of what they do; "A mixin is a way of adding properties and methods of some objects to another object without using inheritance. It allows more complex objects to be created by 'mixing' basic objects together."

This paragraph explains when to use a mixin and when to use inheritance well. : "One way to think about the difference between prototypal inheritance and inheritance from mixin objects is to consider whether an object is something or whether it has something. For example, a tank is a vehicle, so it might inherit from a Vehicle prototype. The tank also has a gun, so this functionality could be added using a gun mixin object. This gives us extra flexibility, since other objects might also use a gun, but not be a vehicle, for example, a soldier object. The soldier object might inherit from a Human prototype and also have the gun mixin."

I can use the return statement: return this; When I do, it allows me to chain methods together on one line. "It's worth keeping in mind that if a method lacks a meaningful return value, it might as well return this so that chaining is possible."

Dealing with global scope when using this:

  • One solution is to set the variable "that" to equal "this" before the nested function.

    superman.findFriends = function(){ const that = this; this.friends.forEach(function(friend) { console.log(`${friend.name} is friends with ${that.name}`); } ); }

  • Another solution is to use the bind() method, here is an example:

    superman.findFriends = function() { this.friends.forEach(function(friend) { console.log(`${friend.name} is friends with ${this.name}`); }.bind(this);)}

  • Alternatively, I could use for-of instead of forEach(), for-of does not require a nested function to be used, so "this" remains bound to the superman object:

    superman.findFriends = function() { for(const friend of this.friends) { console.log(`${friend.name} is friends with ${this.name}`); };}

  • Lastly, I could use an arrow function because they don't have their own "this" context. "So this remains bound to the original object making the function call:"

    superman.findFriends = function() { this.friends.forEach((friend) => { console.log(`${friend.name} is friends with ${this.name}`); } );}

The author advocates "creating small objects that describe single tasks or behaviors" and using a mixin function to combine them into more complex objects if needed.

He also recommends keeping inheritance chains short: "A good rule of thumb is only to inherit once, keeping the inheritance chain to just two objects makes unpicking any issues far easier."

Chapter 15: Modular Javascript

"A module is a self-contained piece of code that provides functions and methods that can then be used in other files and by other modules. This helps to keep code organized in separate, reusable files, which improves code maintainability. The code in a module should have a single purpose, and group together functions with distinct functionality."

  • Importing and Exporting Modules

  • functions in a module have to be exported to be used by another file like this: export const PI = 3.1415926; Then they have to be imported like this:

    import { PI } from './pi.js';

  • I can also export single variables as a default value like this:

    const PI = 3.145926;

    export default PI;

    They are imported like this: import PI from './pi.js';

  • "The big difference with default exports is that you don't need to use curly braces or make any mention of the value that is being imported, making the statement read more elegantly."
